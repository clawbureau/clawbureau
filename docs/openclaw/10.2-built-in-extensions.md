## Purpose

This page surveys the 20+ bundled extensions that ship with OpenClaw. These extensions provide integrations for messaging platforms, specialized tools, memory search backends, and authentication helpers. All bundled extensions live in the `extensions/` workspace directory and are auto-enabled when their configuration is present.

For the plugin system architecture and how extensions are discovered and loaded, see [Plugin System Overview](https://deepwiki.com/openclaw/openclaw/10.1-plugin-system-overview). For building custom plugins, see [Creating Custom Plugins](https://deepwiki.com/openclaw/openclaw/10.3-creating-custom-plugins).

### Discovery Process

Extensions are discovered by the plugin loader during Gateway startup:

1. **Workspace Scan**: The plugin loader scans `extensions/` for directories containing `package.json` with an `openclaw.extensions` field ([extensions/memory-core/package.json 6-9](https://github.com/openclaw/openclaw/blob/bf6ec64f/extensions/memory-core/package.json#L6-L9)).
2. **Manifest Validation**: Each extension's manifest is validated against the plugin schema (requires manifest-embedded config schemas) ([CHANGELOG.md 395](https://github.com/openclaw/openclaw/blob/bf6ec64f/CHANGELOG.md#L395-L395)).
3. **SDK Import**: Extensions import types from `dist/plugin-sdk/` for type safety and interface compliance ([CHANGELOG.md 402](https://github.com/openclaw/openclaw/blob/bf6ec64f/CHANGELOG.md#L402-L402)).
4. **Auto-Enable**: Bundled channel and provider plugins are auto-enabled when their configuration is present ([CHANGELOG.md 406](https://github.com/openclaw/openclaw/blob/bf6ec64f/CHANGELOG.md#L406-L406)).
5. **Registry Insertion**: Validated plugins are inserted into the registry with their declared capabilities (channel, tool, memory, provider slots).

## Configuration and Enablement

### Auto-Enable Behavior

Bundled extensions follow an auto-enable pattern:

| Extension Type | Auto-Enable Condition | Configuration Required |
| --- | --- | --- |
| **Channel** | Config section present | Yes (`channels.<provider>.*`) |
| **Provider** | Model configured | Yes (model selection or auth) |
| **Tool** | Explicit allowlist | Yes (must be listed in `tools.allow`) |
| **Memory** | Provider selected | Optional (defaults to `core`) |

### Example: Enabling MS Teams

```
{
  channels: {
    msteams: {
      // Presence of this section auto-enables the extension
      appId: "...",
      appPassword: "...",
      allowFrom: ["user@example.com"]
    }
  }
}
```

The extension is discovered, validated, and loaded automatically when the Gateway starts.
