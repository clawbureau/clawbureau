## Telegram Native Commands

Telegram commands are registered via the Bot API's `setMyCommands` endpoint. The implementation uses grammY's throttler and sequentialization for rate-limiting and ordered message processing.

### Registration Process

```
Command SourcesRegistration FlowcreateTelegramBot()Build Command ListMerge Custom CommandsValidate No CollisionslistNativeCommandSpecsForConfig()listSkillCommandsForAgents()cfg.channels.telegram.customCommands
```

**Sources:**[src/telegram/bot.ts 348-366](https://github.com/openclaw/openclaw/blob/bf6ec64f/src/telegram/bot.ts#L348-L366) [src/telegram/bot-native-commands.ts 1-100](https://github.com/openclaw/openclaw/blob/bf6ec64f/src/telegram/bot-native-commands.ts#L1-L100)

Telegram commands are set at bot startup:

1. **Command List Building** - `listNativeCommandSpecsForConfig()` retrieves all enabled commands
2. **Skill Commands** - User-invocable skills are automatically registered as `/skill_name` commands
3. **Custom Commands** - `channels.telegram.customCommands` allows additional commands (validated for collisions)
4. **Registration** - Commands sent to Telegram via `setMyCommands()`

### Command Handling

Telegram native commands trigger the `bot.command()` handler, which routes to the unified command processor:

**Flow:** User types `/status` → grammY captures → `registerTelegramNativeCommands()` → `processNativeCommand()` → `dispatchReplyWithDispatcher()`

**Key Implementation Details:**

| Feature | Implementation |
| --- | --- |
| **Sequential Processing** | `sequentialize(getTelegramSequentialKey)` ensures messages from same chat are processed in order |
| **Deduplication** | `createTelegramUpdateDedupe()` prevents duplicate update processing |
| **Argument Parsing** | Commands with args use inline button menus when arguments are missing |
| **Callback Queries** | Button clicks trigger `bot.on("callback_query")` handler |

## Implementation Notes

### Session Keys by Platform

Native commands use isolated session keys to avoid conflicts with text chat sessions:

| Platform | Session Key Pattern | Purpose |
| --- | --- | --- |
| Discord | `agent:{agentId}:discord:slash:{userId}` | Per-user slash command session |
| Telegram | `telegram:slash:{userId}` + CommandTargetSessionKey | Routes to chat session |
| Slack | `agent:{agentId}:slack:slash:{userId}` | Per-user slash command session |

**Exception:**`/stop` command targets the active chat session to abort the current run.

**Sources:**[docs/tools/slash-commands.md 183-190](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/tools/slash-commands.md#L183-L190)

### Access Control

Native commands respect the same access control as text commands:

1. **Allowlist/Policy Check** - `resolveCommandAuthorizedFromAuthorizers()`
2. **Group Policy** - `isDiscordGroupAllowedByPolicy()`, `isSlackChannelAllowedByPolicy()`
3. **DM Policy** - `dmPolicy: "pairing" | "allowlist" | "open" | "disabled"`

Unauthorized command invocations are silently ignored.

**Sources:**[src/discord/monitor/native-command.ts 200-250](https://github.com/openclaw/openclaw/blob/bf6ec64f/src/discord/monitor/native-command.ts#L200-L250) [src/slack/monitor/slash.ts 200-300](https://github.com/openclaw/openclaw/blob/bf6ec64f/src/slack/monitor/slash.ts#L200-L300) [src/telegram/bot-native-commands.ts 50-150](https://github.com/openclaw/openclaw/blob/bf6ec64f/src/telegram/bot-native-commands.ts#L50-L150)
