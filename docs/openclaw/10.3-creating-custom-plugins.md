## Purpose and Scope

This page provides a developer guide for building custom OpenClaw plugins. It covers the plugin manifest structure, SDK usage, and implementation patterns for the four plugin slot types: **Channel**, **Tool**, **Provider**, and **Memory**. For an overview of how plugins integrate into the Gateway architecture, see [Plugin System Overview](https://deepwiki.com/openclaw/openclaw/10.1-plugin-system-overview). For information about the 20+ bundled extensions, see [Built-in Extensions](https://deepwiki.com/openclaw/openclaw/10.2-built-in-extensions).

## Plugin Manifest Structure

OpenClaw discovers plugins by scanning `package.json` files in the `extensions/` workspace directory for the `openclaw.extensions` field. Each plugin must declare its entry point and configuration schema.

### Basic Manifest

The minimal plugin manifest declares an entry point in `package.json`:

```
{
  "name": "@openclaw/memory-core",
  "openclaw": {
    "extensions": [
      "./index.ts"
    ]
  }
}
```

The entry point file must export a plugin definition object that declares the plugin's slot type, configuration schema, and lifecycle hooks.

### Configuration Schema

Plugins must embed their configuration schema using **TypeBox** for runtime validation. The schema is validated during plugin loading, and missing schemas produce warnings.


## Plugin SDK and Types

The Plugin SDK is exported at `dist/plugin-sdk/` and provides TypeScript types, validation utilities, and base interfaces for all slot types.

### SDK Import Paths

| Export Path | Purpose |
| --- | --- |
| `openclaw/plugin-sdk` | Core plugin types and interfaces |
| `openclaw/plugin-sdk/*` | Wildcard access to SDK modules |

The SDK is resolved by the plugin loader at runtime, allowing plugins to use `import type` statements without bundling the entire OpenClaw codebase.

### Core Plugin Interface

Every plugin exports a definition object matching this structure:

```
{
  slot: "channel" | "tool" | "provider" | "memory",
  id: string,
  schema: TypeBoxSchema,
  metadata?: {
    name: string,
    icon?: string,
    description?: string
  },
  init: (config, deps) => Promise<SlotImplementation>
}
```

The `init` function receives the validated configuration and dependency injection context, then returns the slot-specific implementation.

### Channel Slot

Channel plugins integrate messaging platforms. They must implement message routing, access control, and platform-specific features like typing indicators and media handling.

#### Channel Plugin Interface

```
{
  slot: "channel",
  id: "my-channel",
  schema: Type.Object({
    token: Type.String(),
    allowFrom: Type.Optional(Type.Array(Type.String()))
  }),
  init: async (config, deps) => ({
    // Message routing
    start: () => Promise<void>,
    stop: () => Promise<void>,
    send: (target, message) => Promise<void>,
    
    // Access control helpers
    checkAccess: (sender) => boolean,
    
    // Platform features
    setTyping: (target, isTyping) => Promise<void>,
    sendMedia: (target, media) => Promise<void>
  })
}
```

#### Example: Matrix Plugin Structure

The Matrix plugin demonstrates a full channel implementation:

- **Message Routing**: Handles DMs, group mentions, and thread replies
- **Access Control**: Implements allowlist matching and pairing flow
- **Media Handling**: E2EE decryption with size guards
- **Typing Indicators**: Matrix-specific typing state management

### Tool Slot

Tool plugins expose new capabilities to agents. They can be invoked during agent turns and must handle tool call schemas, execution, and result formatting.

#### Tool Plugin Interface

```
{
  slot: "tool",
  id: "my-tool",
  schema: Type.Object({
    enabled: Type.Optional(Type.Boolean())
  }),
  metadata: {
    name: "My Tool",
    description: "Tool description for agent prompts"
  },
  init: async (config, deps) => ({
    // Tool definition for agent
    tool: {
      name: "my_tool",
      description: "What this tool does",
      inputSchema: {
        type: "object",
        properties: {
          param: { type: "string" }
        },
        required: ["param"]
      }
    },
    
    // Tool execution handler
    execute: async (input, context) => {
      // Perform tool action
      return { result: "..." };
    }
  })
}
```

#### Example: Lobster Workflow Tool

The Lobster plugin provides a typed workflow tool with approval gates:

- **Typed Workflows**: JSON-only structured workflows
- **Approval Gates**: Per-step approval prompts
- **File Arguments**: Workflow file args via `argsJson`

### Provider Slot

Provider plugins add AI model inference backends. They implement the model catalog interface and handle authentication, failover, and streaming.

#### Provider Plugin Interface

```
{
  slot: "provider",
  id: "my-provider",
  schema: Type.Object({
    apiKey: Type.Optional(Type.String()),
    baseUrl: Type.Optional(Type.String())
  }),
  init: async (config, deps) => ({
    // Model catalog
    models: [
      {
        id: "model-1",
        provider: "my-provider",
        capabilities: ["chat", "vision"]
      }
    ],
    
    // Streaming inference
    stream: async (model, messages, auth) => {
      // Return AsyncIterable of events
      yield { type: "text", text: "..." };
      yield { type: "done", reason: "stop" };
    }
  })
}
```

Custom providers integrate with the model failover system and auth profile rotation.

### Memory Slot

Memory plugins provide search backends for agent context retrieval. The core memory plugin uses hybrid BM25+vector search with SQLite.

#### Memory Plugin Interface

```
{
  slot: "memory",
  id: "my-memory",
  schema: Type.Object({
    embeddingProvider: Type.Optional(Type.String()),
    storagePath: Type.Optional(Type.String())
  }),
  init: async (config, deps) => ({
    // Indexing
    index: (files) => Promise<void>,
    reindex: () => Promise<void>,
    
    // Search
    search: (query, filters) => Promise<SearchResult[]>,
    
    // Lifecycle
    syncStatus: () => Promise<SyncStatus>,
    close: () => Promise<void>
  })
}
```

The bundled `@openclaw/memory-core` plugin demonstrates:

- **Batch Indexing**: OpenAI batch API for embedding generation

## Plugin Lifecycle

The plugin system follows a discovery â†’ validation â†’ registration â†’ initialization flow. Plugins are auto-enabled when configuration is present.

```
Scan extensions/Load package.jsonValidate schemaConfig presentNo configstart()stop()Schema invalidDiscoveredValidatedInitializedDormantRunningStoppedRejected
```

### Discovery Phase

The plugin loader scans `extensions/` for directories containing `package.json` with an `openclaw.extensions` field. Entry points are resolved relative to the package directory.

**File:**[pnpm-workspace.yaml 1-15](https://github.com/openclaw/openclaw/blob/bf6ec64f/pnpm-workspace.yaml#L1-L15)

### Validation Phase

Each plugin's configuration schema is validated using TypeBox. Missing schemas produce warnings, and invalid schemas reject the plugin.

**Process:**

1. Load entry point module
2. Extract TypeBox schema from plugin definition
3. Validate against config section (if present)
4. Register capability metadata in Plugin Registry

### Initialization Phase

When configuration is present for a plugin's `id`, the Gateway calls `init(config, deps)` with:

- **config**: Validated configuration object
- **deps**: Dependency injection context (logger, storage paths, RPC clients)

The plugin returns its slot-specific implementation, which the Gateway registers with the appropriate subsystem (channel router, tool registry, model catalog, or memory search).

### Auto-Enable Behavior

Bundled channel and provider plugins are auto-enabled when configuration keys exist:

| Plugin Type | Config Key | Auto-Enable |
| --- | --- | --- |
| Channel | `channels.<id>.*` | Yes |
| Provider | `providers.<id>.*` or env vars | Yes |
| Tool | `tools.allowlist` includes plugin tools | Opt-in |
| Memory | `memorySearch.provider` matches `id` | Yes |

Run the Gateway with your plugin:

## Advanced Plugin Patterns

### HTTP Receiver Plugins

Channel plugins can expose HTTP endpoints for webhooks:

```
async init(config, deps) {
  const app = express();
  
  app.post("/webhook", (req, res) => {
    // Handle inbound webhook
    deps.rpc.send({
      method: "message.inject",
      params: { /* ... */ }
    });
    res.status(200).send("OK");
  });
  
  return {
    httpApp: app,
    port: config.webhookPort
  };
}
```

**Example:**[extensions/msteams/](https://github.com/openclaw/openclaw/blob/bf6ec64f/extensions/msteams/) [CHANGELOG.md 390](https://github.com/openclaw/openclaw/blob/bf6ec64f/CHANGELOG.md#L390-L390)

### Cross-Plugin Dependencies

Plugins can declare peer dependencies on other plugins:

```
{
  "openclaw": {
    "extensions": ["./index.ts"],
    "requires": ["@openclaw/memory-core"]
  }
}
```

The loader ensures dependency order during initialization.

### Plugin Commands

Plugins can register CLI commands that appear in `openclaw <plugin-id> <command>`:

```
export default {
  slot: "tool",
  id: "my-plugin",
  commands: [
    {
      name: "sync",
      description: "Sync plugin data",
      handler: async (args) => {
        // Command implementation
      }
    }
  ]
};
```

## Troubleshooting

### Plugin Not Discovered

**Symptom:** Plugin not listed in `openclaw plugins list`

**Solutions:**

1. Verify `openclaw.extensions` field in `package.json`
2. Check entry point path is relative to package directory
3. Ensure plugin is in `extensions/` or installed globally

### Schema Validation Failed

**Symptom:** Warning: "Plugin X missing config schema"

**Solutions:**

1. Embed TypeBox schema in plugin definition
2. Ensure schema matches config structure
3. Check for required fields vs. optional fields

### Plugin Not Auto-Enabled

**Symptom:** Plugin present but not initialized

**Solutions:**

1. Add config section matching plugin `id`
2. For tool plugins, add to `tools.allowlist`
3. For memory plugins, set `memorySearch.provider`
