## Purpose and Scope

This page documents **how configuration is loaded, validated, and managed** in OpenClaw. It covers the CLI commands for reading and writing configuration, environment variable overrides, configuration precedence rules, and validation behavior.

For the **configuration file schema** itself (what keys are available and what they mean), see [Configuration File Structure](https://deepwiki.com/openclaw/openclaw/4.1-configuration-file-structure). For **setting up multiple isolated agents** with separate workspaces and authentication, see [Multi-Agent Configuration](https://deepwiki.com/openclaw/openclaw/4.3-multi-agent-configuration).

## Configuration Sources and Precedence

OpenClaw loads configuration from multiple sources with a clear precedence hierarchy:

```
highest prioritylowest priorityEnvironment Variables
(OPENCLAW_*, ANTHROPIC_API_KEY, etc).env Files
(~/.openclaw/.env)openclaw.json
(~/.openclaw/openclaw.json)Built-in Defaults
(hardcoded fallbacks)Configuration MergeValidated Config
(OpenClawSchema)
```

**Precedence order (highest to lowest):**

1. **Environment variables** - Direct process.env values (e.g., `OPENCLAW_GATEWAY_PORT`, `ANTHROPIC_API_KEY`)
2. **Shell environment** - Variables loaded from dotenv files (`~/.openclaw/.env`)
3. **Configuration file** - `~/.openclaw/openclaw.json` (or `OPENCLAW_CONFIG_PATH`)
4. **Built-in defaults** - Hardcoded fallback values

**Key environment variables:**

- `OPENCLAW_STATE_DIR` - Override state directory (default: `~/.openclaw`)
- `OPENCLAW_CONFIG_PATH` - Override config file path (default: `~/.openclaw/openclaw.json`)
- `OPENCLAW_GATEWAY_PORT` - Override gateway port (default: `18789`)
- `OPENCLAW_GATEWAY_TOKEN` - Set gateway auth token
- `ANTHROPIC_API_KEY` - Anthropic API key (alternative to auth profiles)
- `OPENAI_API_KEY` - OpenAI API key
- Provider-specific keys: `OPENROUTER_API_KEY`, `GEMINI_API_KEY`, `MOONSHOT_API_KEY`, etc.

The gateway service loads `~/.openclaw/.env` automatically on startup via dotenv, ensuring background services have access to credentials without requiring shell initialization.

### openclaw config - Direct Key Manipulation

The `config` command provides low-level access to configuration keys without interactive prompts.

#### Usage Examples

```
# Get a single key (supports dot notation and bracket syntax)
openclaw config get gateway.port
openclaw config get agents.defaults.model.primary
openclaw config get 'channels.whatsapp.allowFrom[0]'

# Set a value (JSON5 parsing for complex values)
openclaw config set gateway.port 19000
openclaw config set gateway.mode local
openclaw config set agents.defaults.model.primary "anthropic/claude-opus-4"

# Set complex values (JSON5)
openclaw config set 'channels.whatsapp.allowFrom' '["+15555550123", "+15555550456"]'

# Unset a key (removes it from config)
openclaw config unset gateway.auth.password
```

#### Behavior

- **Validation**: All writes are validated against `OpenClawSchema` before saving
- **Atomic writes**: Config file is only updated if validation succeeds
- **No prompts**: Designed for scripting and automation
- **Path syntax**: Supports both dot notation (`a.b.c`) and bracket notation (`a["b"].c[0]`)

**Running `openclaw config` with no subcommand launches the interactive wizard** (alias for `openclaw configure`).

**Sources:**[docs/cli/index.md 336-345](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/cli/index.md#L336-L345) [src/config/config.ts 1-15](https://github.com/openclaw/openclaw/blob/bf6ec64f/src/config/config.ts#L1-L15)

### openclaw configure - Interactive Configuration Wizard

The `configure` command provides a **guided wizard** for updating configuration sections.

#### Sections

The wizard organizes configuration into logical sections:

| Section | What it configures |
| --- | --- |
| `models` | Model provider auth, primary model, fallbacks |
| `gateway` | Port, bind mode, auth, Tailscale exposure |
| `channels` | Messaging platform integrations (WhatsApp, Telegram, Discord, etc.) |
| `skills` | Skill installation and node manager selection |
| `web` | Web tool settings (Brave Search API key for `web_search`) |

#### Usage

```
# Launch full wizard (choose sections interactively)
openclaw configure

# Jump directly to a specific section
openclaw configure --section models
openclaw configure --section gateway
openclaw configure --section channels
openclaw configure --section skills
openclaw configure --section web
```

#### Workflow

1. **Section selection** - Choose which areas to configure
2. **Interactive prompts** - Answer questions for each selected section
3. **Config merge** - New values are merged with existing configuration
4. **Validation** - Schema validation runs before writing
5. **Write** - Updated config is saved to `~/.openclaw/openclaw.json`
6. **Gateway restart** - Wizard prompts to restart the gateway if running

The wizard preserves existing values unless explicitly changed, making it safe to run multiple times for incremental updates.

### openclaw doctor - Health Checks and Configuration Repair

The `doctor` command is the **automated repair and migration tool**. It validates configuration, migrates legacy formats, and identifies common misconfigurations.

#### What Doctor Does

```
Phase 4: OutputPhase 3: RepairsPhase 2: MigrationsPhase 1: Validationopenclaw doctorLoad Config
(parseConfigJson5)Validate Schema
(OpenClawSchema)Detect Legacy Keys
(migrateLegacyConfig)Migrate Legacy Config
(routing.* → channels.*)Migrate Legacy State
(sessions, agent dir)Normalize Values
(boolean → object)Check Auth Profiles
(OAuth expiry)Check Sandbox Images
(Docker inspect)Check Gateway Service
(launchd/systemd)Check Security
(open DM policies)Prompt for Repairs
(interactive)Write Updated Config
(writeConfigFile)Show Summary
(notes + warnings)
```

#### Usage Modes

```
# Interactive mode (prompts for fixes)
openclaw doctor

# Non-interactive (only safe migrations, no prompts)
openclaw doctor --non-interactive

# Accept defaults for all prompts
openclaw doctor --yes

# Disable workspace memory hints
openclaw doctor --no-workspace-suggestions

# Deep scan (system-level service detection)
openclaw doctor --deep
```

#### What Gets Checked and Repaired

| Check | What it does | Repair available |
| --- | --- | --- |
| **Config validation** | Validates against `OpenClawSchema` | ✅ Automatic migration |
| **Legacy keys** | Detects deprecated config keys (`routing.*`, `agent.*`) | ✅ Automatic migration |
| **Legacy state** | Detects old session/agent directory layouts | ✅ Prompted migration |
| **Auth profiles** | Checks OAuth token expiry, API key presence | ✅ Refresh expiring tokens |
| **Sandbox images** | Validates Docker images exist when sandbox is enabled | ✅ Recreate missing images |
| **Gateway service** | Checks service config matches current defaults | ✅ Update service config |
| **Gateway auth** | Warns if no `gateway.auth.token` is set (local mode) | ✅ Generate token |
| **Security** | Warns about open DM policies (`dmPolicy: "open"`) | ⚠️ Manual fix |
| **State integrity** | Checks file permissions on config/state directories | ✅ Fix permissions |
| **Model config** | Validates model references, checks auth for providers | ⚠️ Manual fix |

#### Key Repairs

**Gateway token generation:**

```
# Doctor offers to generate and configure a gateway token
# when gateway.auth.token is missing (local mode)
openclaw doctor
# → "Generate and configure a gateway token now? [Y/n]"
```

**Legacy config migration example:**

```
# Before (legacy format)
{
  "routing": {
    "allowFrom": ["+15555550123"],
    "groupChat": { "requireMention": true }
  }
}

# After (migrated by doctor)
{
  "channels": {
    "whatsapp": {
      "allowFrom": ["+15555550123"],
      "groups": { "*": { "requireMention": true } }
    }
  }
}
```

**Exit behavior:**

- Doctor exits with **code 0** even when repairs are needed (unless `--non-interactive` requires explicit scopes)
- Gateway refuses to start when config validation fails; doctor must run first
- Diagnostic commands (`openclaw logs`, `openclaw health`, `openclaw status`) still work with invalid config

## Configuration Validation

All configuration is validated against `OpenClawSchema` (a Zod schema) before being used by the gateway or CLI commands.

### Validation Flow

```
successerrorsopenclaw.json
(raw JSON5)parseConfigJson5
(JSON5 parser)Raw Config ObjectvalidateConfigObject
(OpenClawSchema.safeParse)Valid ConfigBlock Gateway StartSuggest 'openclaw doctor'
```

### Schema Validation Points

| Operation | Validation | Behavior on Error |
| --- | --- | --- |
| `openclaw gateway` | ✅ Full validation | **Refuses to start**, suggests doctor |
| `openclaw config set` | ✅ Full validation | **Rejects write**, shows error |
| `openclaw configure` | ✅ Full validation | **Rejects write**, prompts retry |
| `openclaw doctor` | ✅ Full validation | **Reports issues**, offers migration |
| `openclaw status` | ⚠️ Validation skipped | Runs even with invalid config |
| `openclaw logs` | ⚠️ Validation skipped | Runs even with invalid config |
| `openclaw health` | ⚠️ Validation skipped | Runs even with invalid config |

When validation fails, OpenClaw shows:

1. **Path** - The config key that failed (e.g., `gateway.port`)
2. **Message** - Human-readable error (e.g., "Expected number, received string")
3. **Context** - Surrounding config structure for debugging

**Example validation error:**

```
$ openclaw gateway
Error: Configuration invalid
- gateway.port: Expected number, received string
- channels.whatsapp.allowFrom[0]: Expected string matching E.164 format (+15555550123)

Run 'openclaw doctor' to repair configuration.
```

## Configuration Hot Reload

The gateway watches `~/.openclaw/openclaw.json` for changes and reloads configuration automatically when the file is modified.

### Reload Modes

| Mode | Behavior | When to use |
| --- | --- | --- |
| `hybrid` (default) | Hot-apply safe changes, restart on critical changes | **Recommended** - balance between convenience and safety |
| `hot` | Always hot-apply without restart | Experimental - use with caution |
| `restart` | Always trigger in-process restart | Conservative - ensures clean state |
| `off` | No hot reload (manual restart required) | Debugging, or when stability is critical |

**Configuration:**

```
{
  "gateway": {
    "reload": {
      "mode": "hybrid"
    }
  }
}
```

### What Triggers Restart vs Hot-Apply

**Safe for hot-apply (no restart):**

- Channel-specific settings (`channels.whatsapp.allowFrom`, etc.)
- Message routing settings (`messages.queue`, `messages.groupChat`)
- Tool policies (`tools.allow`, `tools.deny`)
- Agent workspace paths (`agents.defaults.workspace`)
- Sandbox settings (`agents.defaults.sandbox`)

**Requires restart:**

- Gateway bind/port (`gateway.port`, `gateway.bind`)
- Gateway auth (`gateway.auth.mode`, `gateway.auth.token`)
- Plugin configuration changes
- Model provider base URLs (`models.providers.*.baseUrl`)

### Manual Restart

The gateway exposes a **SIGUSR1** handler for in-process restarts:

```
# Find the gateway PID
ps aux | grep "openclaw gateway"

# Send SIGUSR1 to trigger restart
kill -USR1 <pid>
```

The gateway can also be restarted via RPC when `commands.restart` is enabled:

```
openclaw gateway call restart
```

## Multi-Agent Configuration

OpenClaw supports **per-agent configuration overrides** where different agents can have isolated workspaces, authentication profiles, and tool policies.

**Brief overview:**

- `agents.list[]` defines multiple agents
- Each agent can override `workspace`, `agentDir`, `model`, `tools`, `sandbox`
- Routing rules (`bindings`) map channels/accounts to specific agents

**For detailed documentation on multi-agent configuration, see [Multi-Agent Configuration](https://deepwiki.com/openclaw/openclaw/4.3-multi-agent-configuration).**
