## Purpose and Scope

This document specifies the WebSocket RPC protocol used by all clients to communicate with the Gateway Control Plane. The Gateway Protocol defines frame structures, request/response patterns, event streaming, schema validation, and error handling for all client interactions with the gateway server.

For gateway configuration and deployment options, see [Gateway Configuration](https://deepwiki.com/openclaw/openclaw/3.1-gateway-configuration). For managing the gateway as a service, see [Gateway Service Management](https://deepwiki.com/openclaw/openclaw/3.3-gateway-service-management). For remote access patterns, see [Remote Access](https://deepwiki.com/openclaw/openclaw/3.4-remote-access).

## Protocol Overview

The Gateway Protocol is a WebSocket-based RPC protocol that operates on port **18789** (default). All communication flows through three frame types: **RequestFrame** (client → server), **ResponseFrame** (server → client), and **EventFrame** (server → client broadcasts). The protocol enforces strict schema validation using AJV and includes a versioned handshake mechanism.

## Frame Structure

All messages exchanged over the WebSocket connection are **GatewayFrame** objects. The protocol uses a discriminated union of three frame types identified by the `type` field.

### Frame Types

| Frame Type | Direction | Purpose | Schema |
| --- | --- | --- | --- |
| **RequestFrame** | Client → Server | RPC method invocation | `{type: "req", id: string, method: string, params: object}` |
| **ResponseFrame** | Server → Client | RPC method result or error | `{type: "res", id: string, result?: any, error?: ErrorShape}` |
| **EventFrame** | Server → Clients | Broadcast events | `{type: "event", name: string, data: object, ts: number}` |

### RequestFrame

```
{
  type: "req",
  id: string,           // Unique request identifier (client-generated)
  method: string,       // RPC method name (e.g., "agent.run", "sessions.list")
  params: object        // Method-specific parameters
}
```

### ResponseFrame

```
{
  type: "res",
  id: string,           // Matches the RequestFrame.id
  result?: any,         // Success result (mutually exclusive with error)
  error?: ErrorShape    // Error shape (mutually exclusive with result)
}
```

### EventFrame

```
{
  type: "event",
  name: string,         // Event name (e.g., "tick", "agent", "chat", "shutdown")
  data: object,         // Event-specific payload
  ts: number            // Server timestamp (milliseconds since epoch)
}
```

## Schema Validation

All frames undergo strict schema validation using **AJV** before processing. Validation failures result in immediate error responses or connection closure. The protocol exports validation functions for each frame and parameter type.

### Validation Functions

The protocol provides compiled AJV validators for all frame types and RPC method parameters:

| Validator | Purpose | Schema |
| --- | --- | --- |
| `validateConnectParams()` | Handshake parameters | ConnectParamsSchema |
| `validateRequestFrame()` | Request frame structure | RequestFrameSchema |
| `validateResponseFrame()` | Response frame structure | ResponseFrameSchema |
| `validateEventFrame()` | Event frame structure | EventFrameSchema |
| `validateSendParams()` | Message send parameters | SendParamsSchema |
| `validateAgentParams()` | Agent run parameters | AgentParamsSchema |
| `validateSessionsListParams()` | Session list parameters | SessionsListParamsSchema |
| `validateConfigGetParams()` | Config get parameters | ConfigGetParamsSchema |
| `validateCronAddParams()` | Cron job add parameters | CronAddParamsSchema |
| *...and 40+ more* | Method-specific validators | ProtocolSchemas |

When validation fails, the protocol uses `formatValidationErrors()` to produce human-readable error messages:

The protocol defines standard error codes in the **ErrorCodes** enum. All error responses include a `code` field from this enum.

| Error Code | Meaning | Typical Cause |
| --- | --- | --- |
| `INVALID_REQUEST` | Request validation failed | Malformed params, missing required fields |
| `METHOD_NOT_FOUND` | Unknown RPC method | Typo in method name, unsupported method |
| `UNAUTHORIZED` | Authentication required | Missing/invalid token |
| `FORBIDDEN` | Insufficient permissions | Valid auth but denied access |
| `NOT_FOUND` | Resource not found | Invalid session ID, missing config key |
| `CONFLICT` | Resource conflict | Session already exists, duplicate cron job |
| `INTERNAL_ERROR` | Server-side error | Unhandled exception, system failure |
| `TIMEOUT` | Operation timeout | Long-running operation exceeded limit |
| `UNSUPPORTED` | Feature not supported | Disabled feature, unsupported platform |

### ErrorShape

All errors follow the **ErrorShape** schema:

```
{
  code: string,           // ErrorCode enum value
  message: string,        // Human-readable error description
  details?: object,       // Optional additional error context
  stack?: string          // Optional stack trace (debug mode only)
}
```

- `cron.*` - Scheduled job management
- `skills.*` - Skill installation and updates
- `exec.*` - Execution approval management
- `models.*` - Model catalog queries

## Event Streaming

The server broadcasts **EventFrame** messages to all connected clients for real-time updates. Events are fire-and-forget; clients do not acknowledge receipt.

### Event Types

| Event Name | Payload Schema | Purpose |
| --- | --- | --- |
| `tick` | TickEvent | Periodic heartbeat (1s interval) |
| `agent` | AgentEvent | Agent run status updates |
| `chat` | ChatEvent | Chat message streaming |
| `shutdown` | ShutdownEvent | Server shutdown notification |
| `presence` | PresenceEntry | Client presence changes |
| `state_version` | StateVersion | State version updates |

### Tick Events

The gateway broadcasts a **tick** event every second to all connected clients. Clients use this for keepalive, clock sync, and detecting connection health.

```
{
  type: "event",
  name: "tick",
  data: {
    ts: 1234567890000,        // Server timestamp
    uptime: 3600,             // Gateway uptime in seconds
    activeClients?: number,   // Number of connected clients
    activeRuns?: number       // Number of active agent runs
  },
  ts: 1234567890000
}
```

### Agent Events

Agent execution progress is streamed via **agent** events. Each event includes the `runId` and status updates.

```
{
  type: "event",
  name: "agent",
  data: {
    runId: "run-abc123",
    status: "thinking" | "tool_call" | "streaming" | "done" | "error",
    sessionId: "main",
    message?: string,          // Text chunk (streaming)
    toolCall?: {...},          // Tool invocation details
    error?: ErrorShape,        // Error details (if status=error)
    usage?: {...}              // Token usage (if status=done)
  },
  ts: 1234567890000
}
```

### Chat Events

When using the chat RPC methods (`chat.send`), responses stream via **chat** events with block-level structure:

```
{
  type: "event",
  name: "chat",
  data: {
    sessionId: "main",
    blockType: "text" | "thinking" | "tool_call" | "tool_result",
    content?: string,          // Text content (for text blocks)
    thinking?: string,         // Thinking content (for thinking blocks)
    toolCall?: {...},          // Tool call details
    toolResult?: {...},        // Tool result details
    done?: boolean             // True when stream is complete
  },
  ts: 1234567890000
}
```

### Shutdown Events

Before closing connections during shutdown, the gateway broadcasts a **shutdown** event:

```
{
  type: "event",
  name: "shutdown",
  data: {
    reason: "restart" | "signal" | "error",
    message?: string,
    gracePeriodMs?: number     // Time before forced closure
  },
  ts: 1234567890000
}
```

## Authentication

The Gateway Protocol supports three authentication modes configured via `gateway.auth.mode`:

| Mode | Description | ConnectParams |
| --- | --- | --- |
| `token` | Requires `CLAWDBOT_GATEWAY_TOKEN` | `{token: "<secret>"}` |
| `password` | Requires `gateway.auth.password` | `{token: "<password>"}` |
| `tailscale` | Uses Tailscale Serve identity headers | No token required (identity verified server-side) |

### Authentication Flow

### Device Pairing

Clients can also authenticate via **device pairing**. After initial token auth, clients can request device identity tokens that persist across reconnections:

1. Client connects with initial token
2. Client sends `device.pair.request` with `deviceId` and `capabilities`
3. Gateway approves pairing (manually or automatically)
4. Client receives persistent device token
5. Future connections use `{token: "<device-token>"}`

## Protocol Version

The protocol includes a version number (`PROTOCOL_VERSION`) to enable compatibility checks during handshake. Clients receive the server's protocol version in the **HelloOk** response.

**Current Protocol Version:** Defined in [src/gateway/protocol/schema/protocol-schemas.ts](https://github.com/openclaw/openclaw/blob/bf6ec64f/src/gateway/protocol/schema/protocol-schemas.ts) as `PROTOCOL_VERSION = 1`

### Version Compatibility

- Server checks client's expected protocol version (if sent in ConnectParams)
- Client checks server's protocol version in HelloOk
- Mismatched versions should log warnings or reject connections
- Protocol changes that break compatibility should increment `PROTOCOL_VERSION`

## Snapshot State

Clients can request a **Snapshot** of the gateway's current state via the `snapshot` method (implicit, part of connection initialization). The snapshot includes:

```
{
  version: {
    protocol: number,
    gateway: string,
    config: string              // Config file hash/version
  },
  presence: PresenceEntry[],    // Connected clients
  sessions: SessionMetadata[],  // Active sessions
  agents: AgentSummary[],       // Configured agents
  channels: ChannelStatus[],    // Channel connection statuses
  nodes: NodeSummary[],         // Paired nodes
  cron: CronJob[]               // Scheduled jobs
}
```

Snapshots enable clients to quickly reconstruct gateway state after reconnection without querying each subsystem individually.
