This page covers backing up OpenClaw state, restoring from backups, and migrating installations between machines. The goal is to preserve your bot's configuration, credentials, session history, and workspace without redoing onboarding.

**Scope:** This document focuses on the state directory, workspace, and migration mechanics. For initial installation, see [Installation](https://deepwiki.com/openclaw/openclaw/2-installation). For deployment-specific persistence (Docker volumes, cloud storage), see [Deployment](https://deepwiki.com/openclaw/openclaw/13-deployment).

## What Gets Backed Up

OpenClaw stores state in two primary locations:

| Location | Contents | Why It Matters |
| --- | --- | --- |
| **State Directory** (`$OPENCLAW_STATE_DIR`) | Config, auth profiles, sessions, channel state, credentials | Preserves your bot's identity, logins, and conversation history |
| **Workspace** (default: `~/.openclaw/workspace/`) | Memory files, bootstrap templates, agent artifacts | Preserves your bot's knowledge and custom behavior |

Default state directory: `~/.openclaw/` (or `~/.openclaw-<profile>/` for named profiles)

**Sources:**[docs/install/migrating.md 1-190](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L1-L190) [docs/help/faq.md 386-407](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/help/faq.md#L386-L407)

**Key paths:**

## Backup Procedures

```
# Stop the gateway to ensure consistent state
openclaw gateway stop

# Create timestamped archive
BACKUP_NAME="openclaw-backup-$(date +%Y%m%d-%H%M%S).tar.gz"
tar -czf "$BACKUP_NAME" \
  ~/.openclaw \
  ~/.openclaw/workspace

# Restart gateway
openclaw gateway restart

# Verify archive
tar -tzf "$BACKUP_NAME" | head -20
```

**What this captures:**

- All configuration (`openclaw.json`)
- Auth profiles and credentials
- Session history
- Channel state (WhatsApp login, Telegram session)
- Workspace files

**Sources:**[docs/install/migrating.md 66-86](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L66-L86) [docs/platforms/oracle.md 273-281](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/oracle.md#L273-L281) [docs/platforms/digitalocean.md 182-189](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/digitalocean.md#L182-L189)

### Workspace-Only Backup (Git)

If you only want to back up memory and templates (not credentials or sessions):

```
cd ~/.openclaw/workspace
git init
git add .
git commit -m "Backup workspace"
git push origin main
```

**Limitation:** This does **not** preserve:

- Sessions
- Credentials
- Channel logins

Use this for versioning your workspace, not for full disaster recovery.

**Sources:**[docs/install/migrating.md 57-64](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L57-L64) [docs/help/faq.md 399-403](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/help/faq.md#L399-L403)

If you use multiple profiles (e.g., `--profile work`):

```
# Identify all state directories
ls -d ~/.openclaw* | grep -v ".tar.gz"

# Back up each profile
for dir in ~/.openclaw*; do
  if [ -d "$dir" ]; then
    profile=$(basename "$dir")
    tar -czf "${profile}-backup-$(date +%Y%m%d).tar.gz" "$dir"
  fi
done
```

**Sources:**[docs/install/migrating.md 23-47](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L23-L47)

## Restore Procedures

### Full Restore (New Machine)

```
# 1. Install OpenClaw on new machine (if not already installed)
curl -fsSL https://openclaw.bot/install.sh | bash

# 2. Stop any running gateway
openclaw gateway stop

# 3. Extract backup (overwrites existing state)
tar -xzf openclaw-backup-20260129-143022.tar.gz -C ~/

# 4. Run doctor to repair services and apply migrations
openclaw doctor

# 5. Restart gateway
openclaw gateway restart

# 6. Verify
openclaw status
```

**Sources:**[docs/install/migrating.md 87-130](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L87-L130) [docs/help/faq.md 386-407](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/help/faq.md#L386-L407)

### Selective Restore (Config Only)

To restore only configuration without overwriting sessions:

```
# Extract just the config file
tar -xzf backup.tar.gz -C ~/ .openclaw/openclaw.json

# Apply and restart
openclaw doctor
openclaw gateway restart
```
```
# Extract to a named profile
tar -xzf backup.tar.gz
mv .openclaw .openclaw-work

# Run gateway with profile
openclaw gateway --profile work
```

**Sources:**[docs/install/migrating.md 23-47](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L23-L47)

**Sources:**[docs/install/migrating.md 1-190](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L1-L190) [docs/help/faq.md 386-407](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/help/faq.md#L386-L407)

### Step-by-Step Migration

**On the old machine:**

```
# 1. Stop gateway
openclaw gateway stop

# 2. Create backup
cd ~
tar -czf openclaw-state.tar.gz .openclaw
tar -czf openclaw-workspace.tar.gz .openclaw/workspace

# 3. Transfer to new machine
scp openclaw-*.tar.gz user@new-machine:~/
```

**On the new machine:**

```
# 1. Install OpenClaw
curl -fsSL https://openclaw.bot/install.sh | bash

# 2. Extract backups
tar -xzf openclaw-state.tar.gz -C ~/
tar -xzf openclaw-workspace.tar.gz -C ~/

# 3. Fix permissions (if user changed)
chown -R $USER:$USER ~/.openclaw

# 4. Run doctor (repairs services, applies migrations)
openclaw doctor

# 5. Restart gateway
openclaw gateway restart

# 6. Verify
openclaw status
openclaw channels status --probe
```

**Sources:**[docs/install/migrating.md 87-130](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L87-L130)

## Verification Checklist

After restore or migration, confirm:

| Check | Command | Expected Result |
| --- | --- | --- |
| Gateway running | `openclaw status` | Shows "Gateway: running" |
| Config loaded | `openclaw config get gateway.bind` | Shows your bind mode |
| Channels connected | `openclaw channels status` | Shows connected channels |
| Sessions preserved | `ls ~/.openclaw/agents/main/sessions/` | Lists `.jsonl` files |
| Workspace intact | `ls ~/.openclaw/workspace/` | Shows `MEMORY.md`, etc. |
| Auth profiles | `openclaw models status` | Shows auth profiles |

## Common Pitfalls

**Symptom:** Config changes not taking effect, channels missing, empty sessions

**Cause:** Gateway running with a different profile than the one you restored

**Fix:**

```
# Identify active state dir
openclaw status | grep "State dir"

# Ensure it matches your backup location
# If using a profile, specify it:
openclaw gateway stop
openclaw gateway --profile work
```

**Sources:**[docs/install/migrating.md 131-145](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L131-L145)

### Pitfall 2: Partial Copy (Config Only)

**Symptom:** Channels logged out, missing credentials, no session history

**Cause:** Copied only `openclaw.json` instead of the entire state directory

**Fix:** Copy the **entire** `~/.openclaw/` directory, including subdirectories:

- `credentials/`
- `agents/<agentId>/sessions/`
- `workspace/`

**Sources:**[docs/install/migrating.md 146-156](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L146-L156)

**Symptom:** Gateway fails to start with "EACCES" or "permission denied"

**Cause:** Backup restored with wrong ownership (e.g., extracted as root)

**Fix:**

```
# Fix ownership
chown -R $USER:$USER ~/.openclaw

# Restart gateway
openclaw gateway restart
```

**Sources:**[docs/install/migrating.md 157-161](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L157-L161)

### Pitfall 4: Remote vs Local Mode

**Symptom:** Migrated laptop but sessions/workspace missing

**Cause:** UI was in remote mode; state lives on the gateway host, not the UI client

**Fix:** Migrate the **gateway host**, not the client machine

**Sources:**[docs/install/migrating.md 162-167](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L162-L167)

### Pitfall 5: Gateway Lock Persists

**Symptom:** Gateway won't start with "already running" error after restore

**Cause:**`gateway.*.lock` file persists on the new machine but references a dead PID

**Fix:**

```
# Remove stale lock file
rm ~/.openclaw/gateway.*.lock

# Restart gateway
openclaw gateway restart
```

**Sources:**[docs/platforms/fly.md 265-276](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/fly.md#L265-L276)

## Deployment-Specific Persistence

### Docker Volumes

When running in Docker, state persists via volume mounts:

```
volumes:
  - ~/.openclaw:/home/node/.openclaw
  - ~/.openclaw/workspace:/home/node/.openclaw/workspace
```

**Backup Docker state:**

```
# State lives on the host at ~/.openclaw
tar -czf docker-openclaw-backup.tar.gz ~/.openclaw
```

**Sources:**[docs/platforms/gcp.md 199-243](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/gcp.md#L199-L243) [docs/platforms/fly.md 32-77](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/fly.md#L32-L77)

### Cloud VPS (Fly.io, GCP)

Persistent volumes are mounted to `/data` (Fly.io) or host directories (GCP):

**Fly.io:**

```
# SSH into machine
fly ssh console

# Back up /data (where OPENCLAW_STATE_DIR points)
tar -czf /tmp/backup.tar.gz /data

# Download from local machine
fly ssh sftp get /tmp/backup.tar.gz
```

**GCP:**

```
# SSH into VM
gcloud compute ssh openclaw-gateway

# Back up host directories
tar -czf backup.tar.gz ~/.openclaw
```

**Sources:**[docs/platforms/fly.md 1-466](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/fly.md#L1-L466) [docs/platforms/gcp.md 1-499](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/gcp.md#L1-L499)

### Oracle Cloud / VPS

Standard Linux backup applies:

```
# SSH into VPS
ssh ubuntu@openclaw

# Create backup
tar -czf openclaw-backup.tar.gz ~/.openclaw ~/.openclaw/workspace

# Download to local machine
scp ubuntu@openclaw:~/openclaw-backup.tar.gz ~/backups/
```

**Sources:**[docs/platforms/oracle.md 1-292](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/oracle.md#L1-L292) [docs/platforms/digitalocean.md 1-244](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/digitalocean.md#L1-L244) [docs/platforms/exe-dev.md 1-126](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/exe-dev.md#L1-L126)

## Automated Backup Strategies

### Cron Job (Daily Backup)

```
# Add to crontab
crontab -e

# Daily backup at 2 AM
0 2 * * * cd ~ && tar -czf "openclaw-backup-$(date +\%Y\%m\%d).tar.gz" .openclaw && find ~ -name "openclaw-backup-*.tar.gz" -mtime +7 -delete
```

This creates daily backups and deletes archives older than 7 days.

### Git-Based Workspace Backup

For workspace versioning (not credentials):

```
# Initialize workspace repo
cd ~/.openclaw/workspace
git init
git add .
git commit -m "Initial workspace"

# Push to private repo
git remote add origin git@github.com:username/openclaw-workspace.git
git push -u origin main

# Add to crontab for daily commits
0 3 * * * cd ~/.openclaw/workspace && git add . && git commit -m "Auto backup $(date)" && git push
```

**Important:** Do **not** commit the state directory (`~/.openclaw/`) to Git. It contains secrets.

**Sources:**[docs/install/migrating.md 57-64](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L57-L64)

## Security Considerations

### Secrets in Backups

The state directory contains sensitive data:

- API keys (`credentials/`)
- OAuth tokens
- WhatsApp session keys
- Gateway tokens

**Best practices:**

| Practice | Command | Why |
| --- | --- | --- |
| Encrypt backups | `gpg -c backup.tar.gz` | Prevents exposure if backup leaks |
| Restrict permissions | `chmod 600 backup.tar.gz` | Limits access on shared systems |
| Secure transfer | `scp` or encrypted channels | Avoid plaintext over untrusted networks |
| Rotate keys | After suspected exposure | Invalidates leaked credentials |

**Sources:**[docs/install/migrating.md 168-176](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L168-L176)
