This page documents the CLI commands used for troubleshooting, health checks, and system diagnostics. These commands help identify configuration issues, runtime failures, and connectivity problems.

For operational troubleshooting workflows, see [Troubleshooting](https://deepwiki.com/openclaw/openclaw/14.3-common-issues). For configuration management, see [Configuration Commands](https://deepwiki.com/openclaw/openclaw/12.5-configuration-commands). For gateway lifecycle management, see [Gateway Commands](https://deepwiki.com/openclaw/openclaw/12.1-gateway-commands).

## Overview

OpenClaw provides several diagnostic commands, each serving a different diagnostic purpose:

| Command | Purpose | Requires Gateway |
| --- | --- | --- |
| `openclaw status` | Local snapshot: OS, update, gateway reachability, agents, provider config | No (partial) |
| `openclaw doctor` | Health checks, config repairs, state migrations | No |
| `openclaw health` | Gateway health snapshot (RPC) | Yes |
| `openclaw logs` | Stream Gateway logs (RPC) | Yes |
| `openclaw gateway status` | Supervisor runtime state | No |
| `openclaw security audit` | Security posture check | No (partial) |

## Status Command

### openclaw status

Provides a fast local summary without requiring a reachable Gateway. Checks OS info, update availability, gateway reachability, service state, agent configuration, and provider config state.

**Key characteristics:**

- **Fast:** No network timeouts for unreachable Gateways
- **Safe to share:** Redacts tokens (use `OPENCLAW_SHOW_SECRETS=0` for full redaction)
- **Local-first:** Most checks work without a running Gateway
- **Contextual:** Shows which config files are in use

**Output sections:**

1. **System:** OS, OpenClaw version, update availability
2. **Gateway:** Mode (local/remote), reachability, URL, auth status
3. **Service:** Supervisor runtime state (when available)
4. **Agents:** Configured agents, default model
5. **Providers:** OAuth status, API key availability
6. **Sessions:** Active session count (when Gateway is reachable)

**Options:**

```
# Quick snapshot (default)
openclaw status

# Full diagnosis (read-only, safe to share)
openclaw status --all

# Deep probes (requires reachable Gateway)
openclaw status --deep

# Show provider usage/quota
openclaw status --usage

# JSON output
openclaw status --json
```

## Doctor Command

### openclaw doctor

Performs health checks, applies config migrations, and repairs common issues. Doctor is the primary repair tool and should be run after upgrades or when configuration issues are detected.

**What Doctor checks:**

- UI protocol freshness (rebuilds Control UI when needed)
- Config normalization for legacy values
- OAuth credential health and expiry
- Legacy on-disk state migration (sessions, agent directories, WhatsApp auth)
- State integrity and permissions
- Gateway auth configuration
- Sandbox image freshness
- Service configuration drift
- Extra gateway services (with `--deep`)
- Security posture warnings

**Operation modes:**

```
# Interactive (prompts for repairs)
openclaw doctor

# Headless (accept defaults)
openclaw doctor --yes

# Apply recommended repairs
openclaw doctor --repair

# Aggressive repairs
openclaw doctor --repair --force

# Non-interactive (safe migrations only)
openclaw doctor --non-interactive

# Deep scan (system services)
openclaw doctor --deep

# Skip workspace suggestions
openclaw doctor --no-workspace-suggestions
```

## Health Command

### openclaw health

Queries the running Gateway for a full health snapshot via the `health` RPC method. Requires Gateway reachability.

**What it returns:**

- Gateway version and uptime
- Active channel connections
- Agent runtime state
- Session store statistics
- Memory search status
- Sandbox runtime info
- Cron job status
- Recent errors

**Usage:**

```
# Health snapshot (requires reachable Gateway)
openclaw health

# JSON output
openclaw health --json

# Verbose (shows target URL + config path on errors)
openclaw health --verbose

# Custom timeout
openclaw health --timeout 10000
```

**Comparison with `status`:**

- `openclaw status` works without a Gateway; provides local config view
- `openclaw health` requires Gateway RPC; provides runtime state
- `openclaw status --deep` combines both approaches

## Logs Command

### openclaw logs

Streams Gateway logs via RPC. Logs are sourced from the Gateway's file log (`/tmp/openclaw/openclaw-YYYY-MM-DD.log` by default).

**Characteristics:**

- **Requires Gateway RPC:** Cannot read logs if Gateway is unreachable
- **File-backed:** Gateway reads its own log file and streams it
- **Fallback:** If RPC fails, use direct file access: `tail -f /tmp/openclaw/openclaw-*.log`

**Usage:**

```
# Follow mode (live tail)
openclaw logs --follow

# Last N lines
openclaw logs --limit 200

# Plain text (no formatting)
openclaw logs --plain

# JSON output (line-delimited)
openclaw logs --json

# Disable colors
openclaw logs --no-color
```

**Log formats:**

- **TTY sessions:** Colorized, structured view
- **Non-TTY:** Plain text fallback
- **JSON mode:** Line-delimited JSON (one event per line)

**Fallback when Gateway is unreachable:**

```
# Direct file access (latest log)
tail -f "$(ls -t /tmp/openclaw/openclaw-*.log | head -1)"
```

## Gateway Status Command

### openclaw gateway status

Reports supervisor runtime state (launchd/systemd/schtasks) and Gateway RPC reachability. Shows which config the service uses versus which config the CLI uses.

**What it checks:**

- Supervisor state (loaded, running, PID, last exit)
- RPC probe (WebSocket connection + `status` call)
- Config paths (CLI vs service)
- Probe target URL
- Last gateway error (from logs when service looks running but RPC fails)

**Key outputs:**

- **Runtime:** Supervisor state (running, stopped, failed)
- **RPC probe:** Gateway reachability (ok, failed, timeout)
- **Config (cli):** Config file the CLI loaded
- **Config (service):** Config file the service likely uses
- **Probe target:** URL the probe attempted
- **Last gateway error:** Extracted from logs when available

**Usage:**

```
# Status with RPC probe (default)
openclaw gateway status

# Skip RPC probe (supervisor state only)
openclaw gateway status --no-probe

# Deep scan (includes system service scan)
openclaw gateway status --deep

# JSON output
openclaw gateway status --json
```

**Mismatch scenarios:**

When `Config (cli)` ≠ `Config (service)`:

- You're editing one config while the service runs another
- Fix: Reinstall service from correct profile/state-dir: `openclaw gateway install --force`

## Security Audit Command

### openclaw security audit

Audits configuration and local state for security issues. Checks for unsafe defaults, exposed credentials, and permission problems.

**What it checks:**

- Gateway bind mode and auth configuration
- DM policies (pairing, allowlist, open)
- Group mention requirements
- Tool policies (exec, elevated, sandbox)
- State directory permissions
- Config file permissions (should be 600)
- Environment variable exposure
- Tailscale exposure mode

**Usage:**

```
# Basic audit
openclaw security audit

# Deep audit (live Gateway probe)
openclaw security audit --deep

# Auto-fix safe issues
openclaw security audit --fix
```

**Output categories:**

- **Critical:** Immediate security risks (open DMs, no auth, public bind)
- **Warning:** Potentially unsafe configurations (weak DM policies, unsafe tools)
- **Info:** Recommendations (pairing, sandbox, least-privilege)

**Sources:**[docs/cli/index.md 241-246](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/cli/index.md#L241-L246) [docs/gateway/security.md](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/gateway/security.md)
