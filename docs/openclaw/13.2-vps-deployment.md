This page covers deploying OpenClaw on traditional Virtual Private Servers (VPS) from providers like DigitalOcean, Oracle Cloud, Hetzner, and exe.dev. These deployments install OpenClaw directly on a Linux VM using npm or git, with state managed on the host filesystem and systemd supervising the gateway process.

For containerized cloud platforms (Fly.io, Railway, Northflank, GCP with Docker), see [Cloud Deployment](https://deepwiki.com/openclaw/openclaw/13.3-cloud-deployment). For running OpenClaw on your local machine, see [Local Deployment](https://deepwiki.com/openclaw/openclaw/13.1-local-deployment).

**Key architectural points:**

- The Gateway runs as a long-lived process on the VPS host
- systemd ensures automatic restarts on failure or reboot
- State and workspace live on the host filesystem (survives restarts)
- Default binding is `loopback` (127.0.0.1), accessed via SSH tunnel
- Alternative: `lan` binding (0.0.0.0) with token auth, or `tailnet` binding via Tailscale

## Service Management

OpenClaw on VPS uses **systemd user services** for process supervision. The service unit is created during onboarding or via `openclaw gateway install`.

### Service File Location

```
~/.config/systemd/user/openclaw-gateway.service
```

For named profiles:

```
~/.config/systemd/user/openclaw-gateway-<profile>.service
```

### Service Unit Structure

```
[Unit]
Description=OpenClaw Gateway (v<version>)
After=network-online.target
Wants=network-online.target

[Service]
ExecStart=/usr/local/bin/openclaw gateway --port 18789
Restart=always
RestartSec=5
Environment="PATH=/usr/local/bin:/usr/bin:/bin"

[Install]
WantedBy=default.target
```

**Key environment variables** (set in service or via `.profile`):

- `OPENCLAW_STATE_DIR` - State directory (default: `~/.openclaw/`)
- `OPENCLAW_GATEWAY_PORT` - Port (default: 18789)
- `OPENCLAW_GATEWAY_TOKEN` - Auth token for non-loopback binding
- Model provider keys: `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, etc.

**Sources:**[docs/platforms/linux.md 61-90](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/linux.md#L61-L90) [docs/gateway/background-process.md 1-200](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/gateway/background-process.md#L1-L200)

### Common Service Commands

```
# Status
systemctl --user status openclaw-gateway

# Start
systemctl --user start openclaw-gateway

# Stop
systemctl --user stop openclaw-gateway

# Restart
systemctl --user restart openclaw-gateway

# Enable auto-start on boot

# View logs

```

**CLI shortcuts:**

## State and Workspace Management

### Directory Structure

```
~/.openclaw/                      # State directory (OPENCLAW_STATE_DIR)
├── openclaw.json                 # Main configuration
├── credentials/                  # OAuth tokens, API keys
│   ├── anthropic.default.json
│   └── openai.default.json
├── agents/
│   └── main/
│       └── sessions/             # Session history
│           ├── main.jsonl
│           └── dm:telegram:123.jsonl
├── channels/
│   └── whatsapp/                 # Channel state (e.g., WhatsApp login)
│       └── auth_info.json
└── workspace/                    # Agent workspace (default)
    ├── SOUL.md
    ├── USER.md
    ├── MEMORY.md
    └── memory/
        └── snippets.md
```

**State persistence:**

- Config: `~/.openclaw/openclaw.json`
- Credentials: `~/.openclaw/credentials/`
- Sessions: `~/.openclaw/agents/<agentId>/sessions/`
- Channel state: `~/.openclaw/channels/`
- Workspace: `~/.openclaw/workspace/` (or custom location)

All of these survive reboots and service restarts because they live on the host filesystem.

**Sources:**[docs/platforms/digitalocean.md 182-189](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/digitalocean.md#L182-L189) [docs/install/migrating.md 20-76](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L20-L76)

### Backup Strategy

```
# Stop gateway first
openclaw gateway stop

# Archive state + workspace
tar -czf openclaw-state-$(date +%Y%m%d).tgz ~/.openclaw

# Restart gateway
openclaw gateway start
```

**What to back up:**

- `~/.openclaw/` - entire state directory
- `~/.openclaw/workspace/` - workspace (if not in state dir)

**What is ephemeral:**

- PID files: `~/.openclaw/gateway.*.lock`
- Logs: `/tmp/openclaw/openclaw-*.log` (unless configured otherwise)
- Cache: varies by provider

**Sources:**[docs/platforms/digitalocean.md 182-189](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/digitalocean.md#L182-L189) [docs/install/migrating.md 68-86](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/install/migrating.md#L68-L86)

### Binding Modes

**Loopback (recommended):**

```
openclaw config set gateway.bind loopback
openclaw gateway restart
```
- Gateway listens on `127.0.0.1:18789` only
- Not reachable from internet or LAN
- Access via SSH tunnel or Tailscale Serve
- Most secure default

**LAN binding:**

```
openclaw config set gateway.bind lan
openclaw config set gateway.auth.token "$(openssl rand -hex 32)"
openclaw gateway restart
```
- Gateway listens on `0.0.0.0:18789`
- Reachable from LAN (or internet if VPS has public IP)
- Requires strong token or password authentication

**Tailnet binding:**

```
openclaw config set gateway.bind tailnet
openclaw config set gateway.auth.token "$(openssl rand -hex 32)"
openclaw gateway restart
```
- Gateway listens on Tailscale IP:18789
- Accessible from tailnet only
- Requires token/password unless using Tailscale Serve

**Sources:**[docs/platforms/fly.md 79-87](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/fly.md#L79-L87) [docs/platforms/oracle.md 108-125](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/oracle.md#L108-L125) [docs/gateway/configuration.md 1-500](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/gateway/configuration.md#L1-L500)

```
# Forward local port 18789 to VPS loopback

# Then open: http://127.0.0.1:18789/
```

### Tailscale Serve Access

**Setup on VPS:**

```
# Install Tailscale
curl -fsSL https://tailscale.com/install.sh | sh
tailscale up

# Configure Gateway for Serve
openclaw config set gateway.tailscale.mode serve
openclaw config set gateway.bind loopback
openclaw config set gateway.auth.allowTailscale true
openclaw gateway restart
```

**Access:**

```
https://<hostname>.<tailnet>.ts.net/
```

**Key features:**

- HTTPS with automatic certs
- Tailscale identity-based auth (no token needed if `allowTailscale: true`)
- Gateway stays loopback-only on VPS
- No public IP exposure

**Sources:**[docs/platforms/oracle.md 108-125](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/oracle.md#L108-L125) [docs/gateway/tailscale.md 1-200](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/gateway/tailscale.md#L1-L200)

**Key characteristics:**

### Oracle Cloud (Always Free ARM)

**Key characteristics:**

- ARM architecture (aarch64)
- Up to 4 OCPU, 24GB RAM, 200GB storage
- Always Free tier (no charges)
- Capacity/signup can be finicky

**Installation notes:**

```
# ARM-specific: ensure build-essential for native modules
sudo apt install -y build-essential

# Run installer
curl -fsSL https://openclaw.bot/install.sh | bash
```

**Security: VCN + Tailscale pattern:**

1. Lock down VCN ingress rules:
	- Allow: `0.0.0.0/0 UDP 41641` (Tailscale only)
	- Block: all other ports (SSH, HTTP, HTTPS)
2. Gateway stays loopback + Tailscale Serve
3. Connect via tailnet only

This blocks SSH brute force at the network edge while keeping admin access via Tailscale.

**Cost:** $0/month (Always Free)

### Out of memory (OOM)

**Symptoms:** Gateway crashes, silent restarts, OOM in logs

**Fixes:**

1. Add swap (quick fix)
2. Upgrade to larger instance
3. Use lighter models (API-based vs local)
4. Check `NODE_OPTIONS=--max-old-space-size=1536`

**Sources:**[docs/platforms/digitalocean.md 154-177](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/digitalocean.md#L154-L177) [docs/platforms/fly.md 248-263](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/platforms/fly.md#L248-L263)

### SSH tunnel disconnects

**Keep tunnel alive:**

```
# ServerAliveInterval keeps connection active
ssh -o ServerAliveInterval=60 -L 18789:127.0.0.1:18789 user@vps
```

Or use `autossh`:

```
autossh -M 0 -o ServerAliveInterval=60 -L 18789:127.0.0.1:18789 user@vps
```

**Sources:**[docs/gateway/remote.md 1-200](https://github.com/openclaw/openclaw/blob/bf6ec64f/docs/gateway/remote.md#L1-L200)

### State not persisting

**Check state directory:**

```
# Verify location
openclaw status | grep -i state

# Check permissions
ls -la ~/.openclaw/

# Verify config exists
cat ~/.openclaw/openclaw.json
```

**Common causes:**

- Wrong `OPENCLAW_STATE_DIR` in service environment
- Profile mismatch (service runs with different profile than CLI)
- Permissions issue (service user!= login user)

## Security Best Practices

1. **Keep Gateway on loopback:**
	```
	openclaw config set gateway.bind loopback
	```
2. **Access via SSH tunnel or Tailscale Serve**
3. **Strong token authentication:**
	```
	openssl rand -hex 32 > ~/.openclaw/gateway-token
	openclaw config set gateway.auth.token "$(cat ~/.openclaw/gateway-token)"
	```
4. **Regular updates:**
	```
	sudo npm i -g openclaw@latest
	openclaw gateway restart
	```
5. **Backup state regularly:**
	```
	tar -czf openclaw-backup-$(date +%Y%m%d).tgz ~/.openclaw
	```

### If binding to lan/tailnet

1. **Require token or password:**
	```
	openclaw config set gateway.auth.mode token
	openclaw config set gateway.auth.token "$(openssl rand -hex 32)"
	```
2. **Firewall the port:**
	```
	sudo ufw allow from <trusted-ip> to any port 18789
	```
3. **Use Tailscale for VPN access instead of public IP**
4. **Enable HTTPS via reverse proxy (nginx/Tailscale Serve)**
